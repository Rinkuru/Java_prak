# Отчёт по второму этапу

## 1. Разработка классов хранимых объектов

В проекте реализованы четыре Hibernate-сущности:

- `Person` — данные человека, его образование, статус поиска работы и желаемые условия.
- `Company` — данные компании-работодателя.
- `Vacancy` — вакансия компании с требованиями и статусом.
- `WorkExperience` — запись трудовой истории человека.

Связи между сущностями:

- `Person` 1:N `WorkExperience`
- `Company` 1:N `Vacancy`
- `Company` 1:N `WorkExperience`
- `Vacancy` N:1 `Company`
- `WorkExperience` N:1 `Person`
- `WorkExperience` N:1 `Company`

В entity-классах использованы комментарии на русском языке, поясняющие назначение сущности, смысл связи и работу helper-методов.

## 2. Отображение БД на объекты в Hibernate

Отображение реализовано через аннотации `jakarta.persistence` в классах пакета `ru.msu.cmc.java_prak.model`.

Основные решения:

- таблица `person` отображается на `Person`;
- таблица `company` отображается на `Company`;
- таблица `vacancy` отображается на `Vacancy`;
- таблица `work_experience` отображается на `WorkExperience`;
- для полей зарплаты задано `precision = 12` и `scale = 2`, что соответствует `NUMERIC(12, 2)`;
- внешние ключи `company_id`, `person_id` и связанные коллекции описаны через `@ManyToOne` и `@OneToMany`;
- для карточек сущностей используются `join fetch`-запросы, чтобы нужные связи загружались сразу при `spring.jpa.open-in-view=false`.

Отдельный конфигурационный файл Hibernate расположен в `src/main/resources/hibernate.cfg.xml`.

В нём определены:

- подключение к PostgreSQL `hr_agency`;
- `hibernate.dialect`;
- `hibernate.hbm2ddl.auto=validate`;
- явное перечисление классов `Person`, `Company`, `Vacancy`, `WorkExperience`.

Структура БД не изменялась. Hibernate настроен на проверку соответствия существующей схеме, а не на генерацию таблиц.

## 3. Служебные классы DAO

В пакете `ru.msu.cmc.java_prak.dao` находятся интерфейсы DAO, а в пакете `ru.msu.cmc.java_prak.dao.impl` — их реализации на `EntityManager`.

Реализованы следующие DAO:

- `PersonDao` — список людей, карточка человека, CRUD, поиск по образованию, статусу, желаемой зарплате, компании из истории работы, должности из истории работы, комбинированный поиск.
- `CompanyDao` — список компаний, карточка компании, CRUD, поиск по части названия, выборка компаний с открытыми вакансиями, комбинированный поиск.
- `VacancyDao` — список вакансий, карточка вакансии, CRUD, поиск по компании, должности, зарплате и статусу, изменение статуса вакансии, комбинированный поиск.
- `WorkExperienceDao` — CRUD по истории работы, получение истории работы человека, выборка по компании, проверка пересечения периодов работы.
- `MatchingDao` — поиск подходящих вакансий для выбранного человека и поиск подходящих резюме для выбранной вакансии.

Во всех реализациях:

- чтение выполняется в `@Transactional(readOnly = true)`;
- изменяющие методы работают в обычной транзакции;
- нетривиальные запросы оформлены в виде явных JPQL-методов без лишней промежуточной абстракции.

## 4. Модульные тесты TestNG

Тесты написаны на `TestNG` и запускаются через Gradle.

Набор тестов:

- `JavaPrakApplicationTests` — загрузка Spring-контекста;
- `EntityMappingTests` — проверка Hibernate-мэппинга сущностей и связей;
- `EntityHelperTests` — проверка helper-методов синхронизации двусторонних связей;
- `DaoTests` — основные сценарии работы DAO;
- `DaoEdgeCaseTests` — альтернативные ветви и граничные случаи;
- `HibernateConfigurationTests` — проверка наличия и корректности `hibernate.cfg.xml`.

Тестами покрываются:

- CRUD-операции DAO;
- поиск по фильтрам и комбинированные запросы;
- сценарии, когда запись найдена и когда ничего не найдено;
- обе ветви логических флагов сортировки и отбора;
- подбор вакансий и резюме;
- проверка пересечений дат в истории работы;
- helper-методы сущностей с проверкой обеих сторон связи.

Для тестов используется H2 в режиме совместимости с PostgreSQL и отдельная тестовая схема, валидируемая Hibernate.

## 5. Сборка и запуск проверок

В `build.gradle` подключены зависимости для Spring Data JPA, PostgreSQL, H2 и TestNG.  
Тестовая задача Gradle настроена на `useTestNG()`.

Файл `sql/build.xml` дополнен задачами приложения:

- `app-clean` — очистка сборки;
- `app-build` — сборка основного кода;
- `app-test-build` — сборка тестов;
- `app-test` — запуск тестов;
- `app-check` — полная проверка проекта.
